Fase 2 — Hasheo de contraseñas
No necesitas instalar nada externo. .NET incluye BCrypt... bueno, en realidad incluye su propio sistema llamado PasswordHasher, pero la opción más usada y recomendada es el paquete BCrypt.Net.
Instálalo:
bashdotnet add package BCrypt.Net-Next
¿Por qué BCrypt y no MD5 o SHA256? MD5 y SHA256 son rápidos, y eso es malo para contraseñas. Un atacante puede probar millones de combinaciones por segundo. BCrypt es deliberadamente lento y además añade un "salt" automático (un valor aleatorio que se mezcla con la contraseña antes de hashear), por lo que dos usuarios con la misma contraseña tendrán hashes distintos.
El uso es muy simple:
csharp// Al registrar: convertir contraseña a hash
string hash = BCrypt.Net.BCrypt.HashPassword("contraseña_del_usuario");

// Al hacer login: comparar contraseña introducida con el hash guardado
bool esCorrecta = BCrypt.Net.BCrypt.Verify("contraseña_introducida", hashGuardado);
Eso es todo lo que necesitas saber de BCrypt. Lo usarás dentro del servicio de usuarios.

csharppublic class UserService : IUserService
{
    private readonly AppDbContext _context;

    public UserService(AppDbContext context)
    {
        _context = context;
    }

    public async Task<User?> Register(string email, string password)
    {
        // Comprobar si el email ya existe
        if (await _context.Users.AnyAsync(u => u.Email == email))
            return null;

        var user = new User
        {
            Email = email,
            PasswordHash = BCrypt.Net.BCrypt.HashPassword(password)
        };

        _context.Users.Add(user);
        await _context.SaveChangesAsync();
        return user;
    }

    public async Task<User?> Login(string email, string password)
    {
        var user = await _context.Users.FirstOrDefaultAsync(u => u.Email == email);
        if (user == null) return null;

        bool passwordCorrecta = BCrypt.Net.BCrypt.Verify(password, user.PasswordHash);
        if (!passwordCorrecta) return null;

        return user;
    }
}

Fase 4 — JWT (JSON Web Token)
Primero, entiende qué es. Cuando el usuario hace login correctamente, el servidor le devuelve un token, que es una cadena de texto con este aspecto:
eyJhbGciOiJIUzI1NiJ9.eyJlbWFpbCI6InVzZXJAZW1haWwuY29tIn0.abc123...
Tiene tres partes separadas por puntos: cabecera, payload (datos) y firma. El frontend lo guarda y lo envía en cada petición posterior. El backend verifica que la firma es válida y así sabe quién es el usuario, sin necesidad de consultar la base de datos en cada petición.
Instala el paquete:
bashdotnet add package Microsoft.AspNetCore.Authentication.JwtBearer

Configuración del JWT
En appsettings.json añade una sección para el JWT:
json"Jwt": {
  "Key": "una-clave-secreta-larga-minimo-32-caracteres",
  "Issuer": "tu-app",
  "Audience": "tu-app"
}
¿Qué es la Key? Es la clave con la que el servidor firma los tokens. Si alguien la conoce, puede fabricar tokens falsos. Nunca la subas a GitHub. Más adelante la moverás a una variable de entorno Docker.

Servicio para generar tokens
Crea JwtService.cs:
csharpusing System.IdentityModel.Tokens.Jwt;
using System.Security.Claims;
using System.Text;
using Microsoft.IdentityModel.Tokens;

public class JwtService
{
    private readonly IConfiguration _config;

    public JwtService(IConfiguration config)
    {
        _config = config;
    }

    public string GenerateToken(User user)
    {
        var claims = new[]
        {
            new Claim(ClaimTypes.NameIdentifier, user.Id.ToString()),
            new Claim(ClaimTypes.Email, user.Email)
        };

        var key = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(_config["Jwt:Key"]!));
        var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);

        var token = new JwtSecurityToken(
            issuer: _config["Jwt:Issuer"],
            audience: _config["Jwt:Audience"],
            claims: claims,
            expires: DateTime.UtcNow.AddDays(7),
            signingCredentials: creds
        );

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}
¿Qué son los claims? Son los datos que vas a meter dentro del token. En este caso el ID y el email del usuario. Cuando el token llegue en una petición, podrás leer estos datos sin tocar la base de datos.

public class AuthController : ControllerBase

    private readonly JwtService _jwtService;

    public AuthController(IUsuarioService usuarioService, JwtService jwtService)
    {
        _usuarioService = usuarioService;
        _jwtService = jwtService;
    }

    [HttpPost("register")]
    public async Task<IActionResult> Register([FromBody] RegisterDTO dto)
    {
        var user = await _usuarioService.Register(dto.Email, dto.Password);
        if (user == null)
            return BadRequest("El email ya está registrado.");

        var token = _jwtService.GenerateToken(user);
        return Ok(new AuthResponseDTO { Token = token, Email = user.Email });
    }

    [HttpPost("login")]
    public async Task<IActionResult> Login([FromBody] LoginDTO dto)
    {
        var user = await _usuarioService.Login(dto.Email, dto.Password);
        if (user == null)
            return Unauthorized("Email o contraseña incorrectos.");

        var token = _jwtService.GenerateToken(user);
        return Ok(new AuthResponseDTO { Token = token, Email = user.Email });
    }
}
¿Por qué devolver el token al registrarse? Para que el usuario no tenga que hacer login justo después de registrarse. Se registra y queda autenticado directamente, mejor experiencia.

Vamos a seguir el camino completo de una petición de registro, desde que el usuario pulsa "enviar" hasta que el dato llega a la base de datos.

El viaje de una petición de registro
Imagina que el usuario rellena un formulario con email: "raquel@email.com" y password: "1234" y pulsa enviar.

1. El frontend envía una petición HTTP
El navegador manda esto hacia tu backend:
POST https://localhost:7128/api/auth/registrarse
Content-Type: application/json

{
  "email": "raquel@email.com",
  "password": "1234"
}
Es simplemente un mensaje por internet con un destino, un método (POST) y un cuerpo con datos en formato JSON.

2. .NET recibe la petición y la enruta al controlador
.NET tiene un sistema de enrutamiento que lee la URL /api/auth/registrarse y busca qué controlador tiene esa ruta configurada. Encuentra AuthController porque tiene [Route("api/[controller]")] y el método tiene [HttpPost("registrarse")].
¿Por qué [controller] en corchetes? Es un placeholder que .NET sustituye automáticamente por el nombre de la clase sin la palabra "Controller". AuthController → auth.

3. .NET deserializa el JSON en un DTO
El cuerpo JSON que llegó:
json{ "email": "raquel@email.com", "password": "1234" }
Se convierte automáticamente en un objeto DatosUsuarioDTO gracias a [FromBody]. .NET hace ese mapeo solo, comparando los nombres de los campos JSON con las propiedades del DTO.
¿Por qué usamos un DTO y no el modelo Usuario directamente? Porque el modelo tiene campos como Id, CreadoEn, PasswordHash que el usuario no debe enviar. El DTO es exactamente lo que esperamos recibir, ni más ni menos.

4. El controlador llama al servicio
csharpvar user = await _usuarioService.Registro(dto.Email, dto.Password);

¿Cómo sabe el backend quién eres sin consultar la base de datos?
La clave está en entender qué es un token JWT por dentro. No es un código aleatorio como un ticket de vestuario. Es un sobre cerrado con información dentro.
Un token tiene tres partes separadas por puntos:
CABECERA . PAYLOAD . FIRMA
La parte del medio, el payload, contiene los datos que metiste al crearlo:
json{
  "id": "42",
  "email": "raquel@email.com",
  "exp": "2026-04-15"
}
Esto va codificado en Base64, no cifrado. Cualquiera puede decodificarlo y leerlo. Puedes probarlo en jwt.io pegando tu token.
Entonces, ¿qué impide que alguien lo falsifique? La tercera parte: la firma.
Cuando el backend genera el token, coge la cabecera y el payload, y los firma con tu clave secreta (Jwt:Key del appsettings.json):
firma = HMACSHA256(cabecera + payload, "tu-clave-secreta")
Cuando el token llega en una petición futura, el backend repite esa operación. Si la firma que calcula coincide con la del token, significa que el contenido no ha sido modificado y que fue generado por alguien que conocía la clave secreta, es decir, tu propio servidor.
El backend no necesita consultar la base de datos porque la información ya viene dentro del token, y la firma garantiza que es legítima.

¿Cómo se usa el token en la práctica?
El flujo completo es este:
1. Login  → backend devuelve token
2. Frontend guarda el token
3. Cada petición protegida → frontend envía el token
4. Backend verifica la firma → lee los datos → responde
Paso 2 — El frontend guarda el token
En Angular lo guardarías en localStorage:

En Angular esto se hace con un interceptor, que es una clase que se ejecuta automáticamente antes de cada petición HTTP y añade la cabecera. Lo haremos cuando lleguemos al frontend.

¿Qué es .Include(a => a.Imagenes)? Es LINQ de EF Core. Cuando consultas un álbum, EF por defecto no carga sus relaciones. Include le dice "trae también la lista de imágenes". Sin esto, album.Imagenes llegaría vacío y no podrías comparar duplicados.
¿Qué es ToHashSet()? Es una colección de .NET. Convierte la lista de URLs existentes en un HashSet, que tiene búsqueda más rápida que una lista. Luego .Contains(i.Url) comprueba si la URL nueva ya existe.
¿Qué es .Where()? Es LINQ, equivalente al filter() de JavaScript. Filtra solo las imágenes cuya URL no está en las existentes.
¿Qué es .AddRange()? Es un método de List de .NET. Añade varios elementos de golpe en vez de uno por uno con .Add().

import { HttpInterceptorFn } from '@angular/common/http';

export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = localStorage.getItem('token'); // o donde lo guardes

  if (token) {
    const clonedReq = req.clone({
      setHeaders: {
        Authorization: `Bearer ${token}`
      }
    });

    return next(clonedReq);
  }

  return next(req);
};

namespace SketchMuse.Application.Interfaces
{
    public class AlbumesService : IAlbumesService
    {
        private readonly MiDbcontext _context;
        private readonly ImagenesService _imagenesService;

        public AlbumesService(MiDbcontext context)
        {
            _context = context;
        }

        public async Task CrearAlbum(string titulo, int usuarioId, List<ImagenDTO> imagenes)
        {
            var album = new Album
            {
                Titulo = titulo,
                UsuarioId = usuarioId,
                UsedAt = DateTime.UtcNow,
                Imagenes = imagenes.Select(i => new Imagen
                {
                    Url = i.Url,
                    Titulo = i.Titulo
                }).ToList()
            };

            _context.Albumes.Add(album);
            await _context.SaveChangesAsync();
        }

        public async Task AgregarAAlbum(int albumId, int usuarioId)
        {
            var album = await _context.Albumes
                .Include(a => a.Imagenes)
                .FirstOrDefaultAsync(a => a.Id == albumId && a.UsuarioId == usuarioId);

            if (album == null) throw new Exception("Álbum no encontrado.");

            var imagenesNuevas = await _imagenesService.PedirImagenes(album.Titulo, 10);

            // HashSet de URLs existentes para filtrar duplicados
            var urlsExistentes = album.Imagenes.Select(i => i.Url).ToHashSet();

            album.Imagenes.AddRange(
                imagenesNuevas
                    .Where(i => !urlsExistentes.Contains(i.Url))
                    .Select(i => new Imagen { Url = i.Url, Titulo = i.Titulo, AlbumId = album.Id })
            );

            album.UsedAt = DateTime.UtcNow;
            await _context.SaveChangesAsync();
        }

        public async Task<List<AlbumDTO>> GetAlbumesUsuario(int usuarioId)
        {
            var albumes = await _context.Albumes
                .Include(a => a.Imagenes)
                .Where(a => a.UsuarioId == usuarioId)
                .ToListAsync();

            return albumes
                .OrderByDescending(a => a.UsedAt)
                .Select(a => new AlbumDTO
                {
                    Id = a.Id,
                    Titulo = a.Titulo,
                    UsedAt = a.UsedAt,
                    PreviewImagenes = a.Imagenes
                        .Take(3)
                        .Select(i => i.Url)
                        .ToList()
                })
                .ToList();
        }
    }
}

🔍 Qué está pasando
La excepción NullReferenceException significa que estás llamando a un objeto que es null.
En tu controller, la línea 26 es:
var albumes = await _albumService.GetAlbumesUsuario(usuarioId);
Por lo tanto, _albumService es null.
⚠️ Por qué _albumService es null

Miremos tu constructor:

private readonly IAlbumesService _albumService;

public AlbumesController(IAlbumesService albumService)
{
    _albumService = albumService;
}
Si el dependency injection no está registrado correctamente, _albumService nunca se instancia y es null.
Resultado: al llamar a GetAlbumesUsuario → NullReferenceException.
✅ Cómo solucionarlo
1️⃣ Registrar el servicio en Program.cs o Startup.cs

builder.Services.AddScoped<IAlbumesService, AlbumesService>();

builder.Services.AddDbContext<MiDbcontext>(options =>

El error clave es:

GET https://localhost:7128/api/albumes/user-albumes net::ERR_CONNECTION_REFUSED
1️⃣ Qué significa ERR_CONNECTION_REFUSED
Angular intenta llamar a https://localhost:7128/api/albumes/user-albumes
El navegador no puede conectarse al backend en ese puerto.

Posibles causas:

El backend no está corriendo en ese momento.
El backend está corriendo en otro puerto (por ejemplo 5001 o 5000).
Hay un problema de HTTPS / certificado auto-firmado.
2️⃣ Cómo solucionarlo
a) Verifica que el backend esté corriendo
Ve a Visual Studio o tu terminal y asegúrate de que la API esté en ejecución.
Debe mostrar algo como:
Now listening on: https://localhost:7128

Si no, ejecuta tu proyecto backend.

b) Confirma el puerto
En tu launchSettings.json o en la consola de arranque del backend, revisa en qué puerto se está sirviendo.
Si es otro (ej: 5001) → actualiza tu Angular AlbumService:
private apiUrl = 'https://localhost:5001/api/albumes';
c) HTTPS y certificados auto-firmados
Navegadores modernos bloquean peticiones a HTTPS con certificado inválido.
Si es local y certificado auto-firmado:
Abre https://localhost:7128/api/albumes/user-albumes en tu navegador.
Acepta el certificado para confiar en él.
